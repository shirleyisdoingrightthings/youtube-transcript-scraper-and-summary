# 交接文档规范（Handoff Doc · 上下文状态快照）

> **目的**：把"生成"与"多轮精修"两个阶段的上下文**解耦**。每篇稿子在自己的 `output/<标题>/` 目录下，除了「成品.md」「transcript.json」，还要有一份 `交接文档.md`。用户可以**重开一个全新对话**，让新 Agent **只读这份交接文档**（而非重吞整轮生成过程），就能恢复"工作记忆"、接着和用户做后续精修。这是 Token / 上下文管理的核心机制——一页文档替代几十万 token 的历史对话。
>
> **归档三件套**（CLAUDE.md Step 3 强制产出）：`<标题> - <形态>.md`（成品）+ `transcript.json`（字幕，合辑则 `transcript_<来源>.json`）+ `交接文档.md`。

---

## 两条铁律

1. **每篇必出**：初稿交付时（Step 3）即生成 `交接文档.md`，和成品、字幕一起落地，三件套缺一不可。
2. **必须"活"更新**：交接文档不是一次性快照。**每完成一轮精修（Step 4.5）、或定稿后，都要同步更新它的「当前阶段 / ✅已完成 / ⏳待办」**。脱钩的交接文档（成品改了、文档没改）比没有更糟——新 Agent 会据此做出错误判断。改完成品顺手改交接文档，是定式。

---

## 固定模板（按此结构生成，区块顺序不可乱）

````markdown
# 交接文档 ·《<中文标题>》

> **当前阶段**：初稿已交付 / 精修第 N 轮 / 已定稿（可发布）　|　**最后更新**：<YYYY-MM-DD>
> **给接手 Agent 的阅读顺序**：先读本文件 → 再读同目录 `transcript.json`（原始字幕，事实核校的唯一依据）→ 需要时读对应规范 `skills/<illustrated_deepdive 或 dialogue_transcript>.md`。读完即可直接与用户讨论后续精修，**无需回看历史对话**。

## 一、这篇是什么（任务背景）
- **形态**：图文精读 / 对话体逐字稿 / 演讲实录（现场回闪）
- **来源视频**：标题 + YouTube 链接（合辑列全部）+ 字幕覆盖率
- **嘉宾 / 主持人**：一句话身份
- **关键编辑决策**（为什么做成现在这样，避免新 Agent 推翻已定方向）：如"两条视频合成现场回闪一篇""文末不挂 #5 跳转""以本人观点为骨架不引入他人"等。

## 二、关键文件与链接
- 成品：`<标题> - <形态>.md`
- 字幕：`transcript.json`（或 `transcript_<来源>.json`）
- Notion 线上稿：<page URL>　（读回最新线上版：`python3 notion_read.py <page_id_or_url>`）

## 三、✅ 已完成
- 逐条列出已走完的步骤与结论：Step 1 抓字幕（覆盖率）/ Step 2 生成（形态、节数、块数）/ Step 2.5 核校（修了几处硬伤、各是什么）/ Step 2.6 Agent Council（评审分 / 核实分、清了几处）/ Step 3 归档 / Step 4 Notion（链接）。
- 已做的精修轮次与每轮改动要点（随精修推进往下追加）。

## 四、⏳ 待办（接手后优先处理）
- **⚠️ 专名待核**：逐条列出文中标了 `⚠️建议人工核对` 的外文专名 / 事实（位置 + 存疑点 + 已知线索），交用户拍板。
- **🔵 可选润色**：Council 提过、但定为可选的项（如破折号密度、某句译法）。
- **🖼️ 媒体占位**：还需用户补的截图 / GIF（数量、大致位置）。
- **精修方向**：用户已提或可预见的下一步（结构调整、删减、标题钩子、合并外部素材等）。

## 五、给接手 Agent 的一句话
用一句话交代"现在最该和用户确认 / 推进的是什么"。
````

---

## 写作要点

- **写给"冷启动的新 Agent"看**：不依赖任何历史对话，路径、链接、决策都写全，能独立行动。
- **简洁**：交接文档是状态快照、不是成品副本，**绝不复制正文全文**；只写"状态 + 指针 + 待办"。一页左右为宜。
- **待办要可执行**：每条 ⚠️ / 🔵 / 🖼️ 都带"位置 + 是什么 + 怎么处理"，不写空泛的"再检查一下"。
- **阶段标记要准**：顶部「当前阶段」是新 Agent 判断进度的第一信号，每轮务必更新。
- 文件名固定为 `交接文档.md`，放在该稿 `output/<标题>/` 目录下。
