# 对话体逐字稿生成规范

当用户要求生成"对话体逐字稿"（而非深度解读文章）时，必须严格遵守以下原则与结构。

## 核心定位

逐字稿的价值在于**还原对话现场**，让读者能像"旁听者"一样自行提炼观点。它与深度解读文章互补，而非替代。

---

## 编辑原则

**做什么：**
- 准确标注每一段话对应的说话人（主持人 / 嘉宾）
- 在每段对话前标注时间戳，格式为 `[MM:SS]`
- 去掉广告 / 赞助商读广告段落（如 RAMP、WorkOS 等品牌推广内容）
- 去掉纯语气词（如 "um"、"uh"、"you know" 的冗余堆叠）和过于口语化的填充表达
- 修正明显的语法错误和自动听写误差（如人名、公司名、术语拼写）
- 保留对话的自然节奏——追问、打断、笑声提示等可简短标注（如 `[笑]`）

**不做什么：**
- 不重构对话顺序——严格按照原始时间线排列
- 不删除"看似不重要"的对话——读者需要完整上下文来自行判断价值
- 不替换嘉宾用词——即使嘉宾的表述不够"学术化"，也要保留原意
- 不插入编辑评论或注解——逐字稿是中性记录，不是解读
- 不将对话内容改写为段落体——保持一问一答的对话格式

---

## 说话人识别规则

1. **从视频标题、频道信息和对话内容推断说话人身份**
2. 字幕中的 `>>` 符号标记说话人切换
3. 通常提问者 = 主持人，回答者 = 嘉宾
4. 在文档开头明确列出所有说话人及其身份

---

## 输出结构（严格按照以下格式生成）

逐字稿文档由以下四个部分组成，顺序不可调换：

1. **YAML Frontmatter** — 标签声明
2. **背景信息** — 嘉宾介绍（自然段落，不加标题）
3. **访谈信息**（`## 访谈信息`）— 结构化元数据列表
4. **TL;DR + 核心观点**（`## TL;DR` / `## 核心观点`）— 导读层，与深度解读文章格式一致
5. **逐字稿正文** — 按时间戳顺序还原的对话原文

---

### 文档头部（YAML Frontmatter）

```markdown
---
tags:
  - 对话访谈
  - <根据内容从 tags.json 中选取 1-2 个额外标签>
type: dialogue-transcript
---
```

### 背景信息（YAML 之后，`## 访谈信息` 之前）

不加任何二级标题，以 **1 段自然段落** 写入嘉宾背景介绍：
- 介绍嘉宾的身份、所属机构、核心专长领域
- 说明嘉宾为什么有资格讨论本期话题（独特视角、行业地位、一手经验等）
- 所有信息必须来源于字幕内容，不得自行编造嘉宾履历

### 访谈信息（`## 访谈信息`）

```markdown
## 访谈信息

**视频标题**：<原始英文标题>

**主持人**：<姓名，职位/节目名>

**节目嘉宾**：<姓名，职位，所属机构>

**视频链接**：<URL>

**播出时间**：<时间>

**逐字稿说明**：本文为该访谈的对话体逐字稿，已去除广告段落和冗余语气词，修正了听写误差，保留对话原始顺序与说话人风格。
```

### TL;DR（`## TL;DR`）

与深度解读文章的 `## 2. TL;DR` 格式完全一致：

```markdown
## TL;DR

<150-200 字。用一段话概括本期访谈的核心问题、主要结论、关键判断，以及值得关注的分歧或不确定性。适合读者在决定是否深读全文前快速扫读。>
```

### 核心观点（`## 核心观点`）

与深度解读文章的 `## 3. 核心观点` 格式完全一致。**不设固定条数上限**，根据内容丰富程度灵活决定：
- 短访谈或聚焦型内容：3-5 条
- 长访谈或多主题内容：可写至 8-10 条
- 原「重点关注」中提炼的内容可直接并入核心观点，不单独列出

每条格式（注意使用标准 Markdown 数字列表，标题加粗，正文提行并缩进）：

```markdown
## 核心观点

1. **<观点标题>**
   <2-3 句展开说明，信息密度高，可独立引用>

2. **<观点标题>**
   <2-3 句展开说明>

...（条数根据内容决定，无硬性上限）
```

### 逐字稿正文

在核心观点之后，用分割线隔开，进入逐字稿正文。**此分割线之后不再有任何导读内容，只有按时间戳顺序排列的对话原文。**

```markdown
---

## 以下为逐字稿正文：

**[00:30] 主持人：** 你告诉过我一个关于你们团队今年 Token 使用量剧变的故事。能再讲一次吗？它教会了你什么？

**[00:41] Dylan：** 去年我们以为自己已经是 AI 的重度用户了。每个人都在用 ChatGPT、Claude，我给全公司买了各种订阅，花费大概在几万美元的量级……
```

- 在分割线后，正文开始前添加二级标题 `## 以下为逐字稿正文：`
- 每段对话以 `**[MM:SS] 说话人：** ` 开头（注意标点统一使用中文冒号，且冒号后只保留一个空格）
- 同一人连续说的长段落可以保持为一整段，不需要逐句换行
- 当说话人切换时，空一行



## 标题生成规则

与深度解读文章使用相同的标题生成规则（详见 `article_generation.md` 中的「标题生成规则」章节）。

**格式：** `嘉宾姓名｜核心主题描述`

---

## 文件命名与存放

- 文件名格式：`<生成的中文标题> - 逐字稿.md`
- 存放在与深度解读文章相同的子目录下：`output/<生成的中文标题>/`
- 同一期视频最终目录下应包含三个文件：
  - `<生成的中文标题> - 深度文章.md`（深度解读文章，如已生成）
  - `<生成的中文标题> - 逐字稿.md`（对话体逐字稿）
  - `transcript.json`（原始字幕数据）

---

## 输出语言

- **全部翻译为简体中文**。无论原视频是什么语言，最终输出的逐字稿内容必须是中文。
- **说话人标签规则**：第一次出现说话人时，使用完整姓名（如 Patrick O'Shaughnessy、Dylan Patel）；之后再次出现时，使用名字（如 Patrick、Dylan）。
- **行业专有名词处理**：
  - 保留高共识词汇的英文原貌（例如 Token、Agent、GPU、API、Prompt 等）。
  - 针对较新的技术词汇或特定行业名词，在第一次出现时，请在后面括号标注中文解释。
