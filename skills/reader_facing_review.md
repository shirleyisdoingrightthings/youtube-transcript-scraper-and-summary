# 面向读者的精编稿二次复核规范（Reader-Facing Review）

> **调用方式**：当用户把一段（或几段）已成形的逐字稿 / 精编稿发来，要求"复核""二次复核""帮我看看这段有没有问题""对照字幕检查"时，加载并执行本规范。
> **定位**：本规范处理的是**面向读者发布的精编稿**（公众号 / Blog），**不是严格逐字稿**——因此"为读者落地而做的增补"是加分项，而非违规。
> **配套**：生成阶段用 `dialogue_transcript.md` / `illustrated_deepdive.md`；术语统一始终查 `glossary.md`；本规范用于生成之后的逐段打磨。

---

## 执行顺序（固定四步 + 产出）

每次复核按 A→B→C→D 跑一遍，最后给"完整优化版"。**先报问题、再改稿**，不要把问题藏在改稿里让用户自己找。

### A. 对照字幕核校（忠实度 + 语言表述）

1. 打开该段对应的 `transcript.json`（原始字幕），逐句比对译文。
2. **区分三档，分别处理**：
   - 🔴 **必改硬伤**：事实/数字/专名错误、逻辑反转、**意思译反或译丢**（典型：把 "have conversations with folks I support" 误成"安排会议"；catch a bug 误成"引出新 bug"）。逐处列出"原文→你的稿→正确译法"。
   - 🟡 **轻微偏差**：语义略漂、漏掉招牌词/关键限定（如 trust but verify、high accountability）。给出建议但标为可选。
   - 🔵 **可选润色**：语序、节奏、措辞，由用户定。
3. **专名核对铁律**：外文公司/产品/模型/人名，先查证官方写法（WebSearch/WebFetch 或用户随稿提供的参考清单），查不到就标 `⚠️建议人工核对`，绝不照搬音译猜测。
4. **每段给总判定**：`✅ 本段无硬伤` 或 `⚠️ 本段 N 处硬伤`，不让用户猜。

### B. 标题诊断（标题准不准 + 标题与内容是否对应 + 分段是否合理）

1. **标题要具体**：禁用功能性/描述性标题（❌"工程师角色的转变""关于 X 的讨论"）。标题要短，但必须点出这节到底讲什么，最好直接是一句有判断力的钩子。
2. **标题↔内容必须对应**：核对每节标题下的内容是否真的在讲这个标题。**对照字幕时间线检查标题是否"挂错了内容"或"整体错位一位"**（高频错误：复制粘贴时标题与内容错位）。发现错位，明确指出并重新对齐。
3. **分段是否合理**：一节只讲一个主题。若一节里塞了两拍（如"管理者掌握全局"+"验证成新瓶颈"），建议**拆成两节**或补一句显式过渡。相邻同主题碎节则建议**合并**。精编稿一篇约 8 节为宜。
4. 给出"建议标题对照表"（你的标题 → 实际内容 → 建议标题）。

### C. 面向读者的增补（帮非技术 / 非 Claude Code 用户落地）

1. **这是精编稿、不是严格逐字稿**，可以为读者补充背景，让内容更易懂。增补的两类对象：
   - 不熟悉 Claude Code / Cowork 的读者：解释功能是什么、解决什么问题。
   - 非技术读者：解释 repo、PR、TDD、Spec、cron job、分布式系统、async 等术语。
2. **增补必须与嘉宾原话明确区分**：统一用灰框（`> 🔍 名词解释 · …`）或显式"编者注"，绝不混入说话人发言冒充原话。
3. **Claude Code / Cowork 功能解释必须准确**（常备口径，拿不准先查证再写）：
   - **Routines**：定时自动化，但比 cron job 多一层——会 prompt agent、直接生成 PR 供审、发现 bug 可直接修。"到点派一个会思考的助手把活办完"，而非"到点跑一条死命令"。
   - **Skills**：把一套标准/做法打包成可复用指令，放进仓库供 Claude 反复调用。
   - **Spec**：规范文档，"什么算对/好"的书面标准；check 进仓库与代码一起更新。
   - **Code Review（自动代码评审）**：合并前让 Claude 对照规则挑问题、给建议。
   - **常驻远程会话**：长期挂云端、能读所有 repo/Slack/指标的 Claude 实例，相当于管理者的"全局仪表盘"。
   - **PR / repo / dogfooding / agent 舰队 / async / latent demand / agency**：按 glossary 与通用口径简释。
4. **增补是加分不是改原意**：只补背景、不替嘉宾下新结论；补充的事实若无出处，标 `⚠️建议人工核对`。

### D. 篇幅与取舍（嘉宾画像决定详略）

1. **结合嘉宾身份判断详略**：资深高管 / 一线负责人，读者更想看**结合应用的实例、方法论、技术判断**，而非大段个人前史。
2. **倾向保留**：关键技术判断、产品/管理实例、可独立引用的金句、可复制的方法论、数据。
3. **倾向压缩或删除**：冗长的个人成长背景故事、纯铺垫长问、重复论证、绕回去又说一遍的内容、广告口播。
4. 删除某节时，检查**它是否是后文的铺垫**；若是，把后文里依赖它的那句改写成不依赖前文的版本，保证衔接不断。

## 产出格式

1. **核校与诊断报告**（先给）：A 的硬伤清单（🔴/🟡/🔵）+ 总判定；B 的标题诊断表；C 的增补建议；D 的取舍建议。最关键的问题放最前。
2. **完整优化版**（后给）：按上述建议改好的整段，含增补灰框、对齐后的标题、修正后的译文。
3. **去 AI 味自检**：定稿前扫一遍——少用破折号、慎用"不是…而是…"、删包装词与空转连接（详见 `dialogue_transcript.md` 去 AI 味章节）。
4. 若涉及永久性规范变更，按 CLAUDE.md Step 6 Meta-Harness 提示用户确认后再落文档。

## Agent Council 终审协议（两个关口共用）

本工作流在两处用 **Agent Council** 做互审（详见 CLAUDE.md 的 Step 2.6 交付前自检、Step 4.5 定稿前终审）：

**机制（三角色）：**
1. **评审 Agent**（编辑视角，全新 Agent）：对照字幕 / 完整正文打分（总分 + 维度分）+ 出"位置 + 问题 + 改法 + 严重度"清单。
2. **核实 Agent**（审稿视角，另起全新 Agent）：拿**完整正文**逐条判定评审清单 真实 / 部分属实 / 不成立 + 是否必要改，并补评审漏掉的问题。
3. **协调方（主模型）**：综合两方共识，先把必改硬伤清零，再向用户汇报"可发布"判定与分数。

**两条铁律：**
- **必须是全新 Agent**——模型不能给自己打分（生成稿的模型不得自评）。
- **必须喂完整正文，绝不喂压缩摘要**——实测教训：曾因把正文压成"分号串要点"喂给评审 Agent，导致它误判"分号流水账 / 可读性崩塌"，靠第二个 Agent 拿真稿才纠偏。压缩稿会制造假阳性。

**读回线上稿**：若稿子已在 Notion，用 `python3 notion_read.py <page_id_or_url>` 把完整正文读回本地再喂给 Council。注意：页面被移出上传数据库后集成会返回 404，需在 Notion 把该页重新「连接（Connections）」给集成 `claude code youtube transcript scraper and summary` 才能读写。
